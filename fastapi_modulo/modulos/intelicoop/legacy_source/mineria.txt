Minería de datos

Módulo de Minería de Datos para Cartera de Crédito
Objetivo del Módulo
Ejecutar procesos de extracción de conocimiento (modelos predictivos y segmentaciones) sobre los datos de la cooperativa para generar valor: predecir riesgo, detectar patrones de mora y encontrar oportunidades de negocio.

Elementos MAIN para Implementación del Módulo de Minería de Datos

1. Diagnóstico Inicial
- El módulo debe arrancar con foco en crédito y riesgo.
- La prioridad operativa es generar resultados consumibles por API y guardarlos históricamente.
- El diseño debe soportar ejecución en tiempo real (scoring) y en batch (mora, segmentación, asociaciones).

2. Brechas a Definir Antes de Implementar
- Definir etiqueta de incumplimiento: 30, 60 o 90 días de mora.
- Definir periodicidad de reentrenamiento por submódulo.
- Definir umbrales de negocio: aprobar, evaluar, rechazar.
- Definir trazabilidad de modelos: versión, fecha de entrenamiento, dataset usado.
- Definir política de monitoreo de drift y desempeño.

3. Plan de Implementación por Fases (MVP)
- Fase 1: Scoring crediticio online + almacenamiento de resultados.
- Fase 2: Predicción de mora temprana (30/60/90 días) en proceso batch.
- Fase 3: Segmentación mensual de socios con perfiles interpretables.
- Fase 4: Reglas de asociación entre productos para oportunidades comerciales.

4. Tablas Mínimas del Repositorio Analítico
- mineria_model_version:
  nombre_modelo, version, algoritmo, fecha_entrenamiento, metricas, estado.
- mineria_feature_snapshot:
  id_entidad, fecha_corte, variables usadas en inferencia.
- mineria_resultado_scoring:
  id_solicitud, score_crediticio, probabilidad_incumplimiento, recomendacion, version_modelo, fecha.
- mineria_alerta_mora:
  id_credito, prob_mora_30d, prob_mora_60d, prob_mora_90d, nivel_alerta, fecha.
- mineria_segmento_socio:
  id_socio, segmento, fecha_ejecucion, version_modelo.
- mineria_regla_asociacion:
  regla, soporte, confianza, lift, fecha_ejecucion.

5. APIs Mínimas del Módulo
- POST /api/ml/scoring
- GET /api/ml/scoring/{id_solicitud}
- GET /api/ml/mora-alertas?dias=30|60|90
- GET /api/ml/segmentos/socios
- GET /api/ml/reglas-asociacion

6. Información que Necesitamos Subir al Módulo

6.1 Datos de Socios
- id_socio, edad, género, estado civil, nivel educativo, actividad económica, zona geográfica.
- fecha_alta_socio, antigüedad, canales usados (app/backend/oficina).

6.2 Datos de Créditos
- id_credito, id_socio, fecha_desembolso, monto, plazo, tasa, destino, garantías.
- estado_credito, fecha_vencimiento, saldo_actual.

6.3 Datos de Pagos y Mora
- id_pago, id_credito, fecha_vencimiento, fecha_pago, monto_pagado.
- días_atraso por cuota, mora acumulada, reestructuraciones.

6.4 Datos de Ahorros y Transacciones
- saldo_promedio, saldo_minimo, saldo_maximo.
- frecuencia de depósitos/retiros/transferencias.
- total movimientos por periodo.

6.5 Datos Externos (Opcional pero Recomendado)
- score de buró, endeudamiento global, consultas recientes en centrales de riesgo.
- variables macro y demográficas por zona.

7. Formato de Carga Recomendado
- Batch diario: CSV o Parquet para tablas históricas.
- Tiempo real: JSON por API para scoring en originación.
- Clave de integración obligatoria: id_socio e id_credito consistentes entre sistemas.

8. Reglas de Calidad de Datos (Obligatorias)
- No permitir IDs nulos o duplicados.
- Validar rangos: monto > 0, tasa >= 0, plazo > 0.
- Homologar catálogos: estados de crédito, tipos de producto, canales.
- Registrar fecha_corte en toda carga analítica.

9. Criterios de Aceptación del Módulo
- Respuesta de scoring menor a 500 ms en ambiente productivo.
- Resultados batch guardados con versionado y fecha de ejecución.
- Trazabilidad completa de modelo y variables utilizadas.
- Endpoints disponibles para consumo por originación, cobranza y dashboard.
